# 📋 Process Area — Requirements

> This folder is **self-contained**: everything you need to understand the area is here.

## 📌 What this area is in MPS.BR
Covers requirements capture, specification and management. In MR-MPS-SW it maps to **Requirements Management (GRE)**: keeping agreed requirements traceable.

## 🎯 What problem it solves
Solves **ambiguous and shifting scope**, the root cause of most rework.

## ✅ What belongs in this area
- Functional and non-functional requirements
- Traceability matrix
- Scope change requests
- Business rules and acceptance criteria

## 🚫 What does NOT belong in this area
- How to build it (→ architecture)
- Schedule (→ projects)
- Test cases (→ quality)

## 📈 How it evolves from level G to A

| Level MPS.BR | Evolution of the area |
|---|---|
| Level G — Partially Managed | Documented and managed requirements. |
| Level F — Managed | Requirements under formal change control. |
| Level E — Partially Defined | Standardized requirements process. |
| Level D — Largely Defined | Integrated into the organizational process. |
| Level C — Defined | Requirements linked to decisions and reuse. |
| Level B — Quantitatively Managed | Volatility measured quantitatively. |
| Level A — Optimizing | Causal analysis reduces requirements rework. |
