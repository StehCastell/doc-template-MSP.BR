# Lessons Learned

## Objective

This document aims to record knowledge acquired during project execution, allowing the identification of good practices, improvement opportunities and preventive actions for future projects.

Lessons learned contribute to:

- Continuous process improvement;
- Reduction of recurring errors;
- Knowledge sharing;
- Evolution of development practices;
- Support for future decision-making.

---

# When to Record a Lesson Learned

A lesson learned may be recorded when the following occurs:

- Completion of a release;
- Closure of a project phase;
- Resolution of a relevant problem;
- Identification of a good practice;
- Identification of a materialized risk;
- Occurrence of significant failures or rework;
- Implementation of process improvements.

---

# Categories

Lessons learned can be classified into the following categories:

| Category | Description |
|------------|------------|
| Requirements | Elicitation, analysis and requirements management |
| Planning | Schedule, estimates and tracking |
| Development | Implementation and architecture |
| Quality | Testing, review and validation |
| Processes | Workflows and governance |
| Communication | Interaction between stakeholders |
| Infrastructure | Environments and tools |
| Risks | Risk management and mitigation |
| Deployment | Product publishing and delivery |

# Best Practices

- Record the lesson as close as possible to the event.
- Describe facts objectively.
- Avoid assigning personal responsibility.
- Focus on processes, practices and results.
- Record both successes and failures.
- Periodically review the recorded lessons.
